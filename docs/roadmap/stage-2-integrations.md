# Этап 2: Интеграции И Работающие Связи

Цель: перевести интерфейс из демо-режима в рабочий режим с реальными данными и доставкой заявок.

## 1) Форма контакта (критично)

Проблема:
- `src/components/sections/ContactSection.tsx` использует `setTimeout` вместо реальной отправки.

Что реализовать:
- API endpoint для отправки заявки (`POST /api/leads` или webhook-адаптер).
- Валидацию payload на бэкенде и фронтенде.
- Нормальные статусы UI: `idle/loading/success/error`.
- Логирование ошибок и защиту от дублей (idempotency key / debounce).

Definition of Done:
- Заявка реально уходит во внешний канал (CRM/таблица/почта/Telegram).
- Пользователь видит корректный результат, а не имитацию.

## 2) Интеграционный слой (данные и связи)

Проблема:
- `src/integrations/supabase/types.ts` содержит пустую схему (`Tables: never`).
- Supabase клиент не подключен к реальным сценариям UI.

Что реализовать:
- Определить источник истины для лидов и справочников (Supabase или другой backend).
- Актуализировать типы БД.
- Вынести вызовы в сервисный слой (`src/services/*`) и подключить через React Query.

Definition of Done:
- У интерфейса есть реальные `query/mutation`, не демо-данные.

## 3) Вебхуки и внешние получатели

Проблема:
- В коде нет настроенных webhook-интеграций.

Что реализовать:
- Единый модуль отправки в webhook (retry + таймаут + обработка 4xx/5xx).
- Конфиг через env (`VITE_*` только для публичных параметров, секреты на сервере).
- Таблица/лог доставки для технического контроля.

Definition of Done:
- Каждая заявка имеет trace: `accepted -> delivered` или `accepted -> failed (с причиной)`.

## 4) Технический долг по UI-хукам

Проблема:
- Есть дубли/наследие shadcn-обвязки, часть модулей не используется в текущем потоке.

Что реализовать:
- Консолидацию toast-слоя и единый импорт.
- Удаление неиспользуемых оберток после подключения реальных флоу.

Definition of Done:
- Нет “мертвых” хуков/оберток в основном пользовательском пути.
